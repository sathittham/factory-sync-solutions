# Domain Event Contract

## Purpose

This document defines the event envelope and behavior for the event-driven evolution
from a modular monolith to Cloud Run domain services.

It is the contract that all future asynchronous consumers must follow before we
split services.

## Canonical Event Envelope

All emitted domain events must conform to this envelope:

```json
{
  "id": "uuid",
  "type": "ProfileRegistered|ProfileUpdated|QuizSubmitted|ResultReady",
  "aggregateType": "profile|quiz|assessment",
  "aggregateId": "uuid",
  "actorUID": "firebase-user-id",
  "correlationId": "uuid|request-id|trace-id",
  "payload": {},
  "metadata": {},
  "occurredAt": "2026-06-18T00:00:00Z"
}
```

- `id`: unique event UUID generated by the producer.
- `type`: stable, append-only event type.
- `aggregateType`: domain entity type (`profile`, `quiz`, `assessment`).
- `aggregateId`: source domain id (`userID`/`uid`/`assessmentId`).
- `actorUID`: authenticated actor id (`middleware.GetUID(r)` on the producer path).
- `correlationId`: request trace propagation key; use `x-request-id` when available.
- `occurredAt`: RFC3339 UTC timestamp, set by producer at emit time.

## Event Types (Phase 1)

| Event | Producer | Aggregate | Payload Minimum |
|---|---|---|---|
| `ProfileRegistered` | `profile` | `profile` | `uid`, `companyRegId` |
| `ProfileUpdated` | `profile` | `profile` | `uid`, `fields` |
| `QuizSubmitted` | `quiz` | `quiz` | `assessmentId`, `quizId`, `overallScore` |
| `ResultReady` | `quiz` | `assessment` | `assessmentId`, `uid`, `quizId`, `overallScore` |

Current contract validation checks:
- event `type` must be one of the four canonical values
- required header fields (`id`, `aggregateType`, `aggregateId`, `occurredAt`) must be set
- required payload fields listed above must be present

### Event payload rules

- payload keys use `camelCase`.
- include only values needed by downstream consumers.
- keep payload stable per version; adding fields is allowed, removing/renaming is not.
- keep PII to minimum needed for business flow.

## Producer behavior (current phase)

1. Persist state in Firestore transaction or post-commit write path.
2. Emit event only after successful persistence.
3. Never block request lifecycle for non-essential downstream side effects.
4. Failure to emit in `log` mode is non-blocking; in future strict mode it should
   be surfaced to monitoring with retry path.

## Consumer behavior (future phase)

1. Consumers must be idempotent: at-least-once delivery is expected.
2. Store and check dedupe keys by `event.id` and `aggregateType:aggregateId`.
3. If dedupe check fails, treat the event as already processed and acknowledge safely.
4. On transient failures, retry with backoff.
5. On terminal failure, publish to DLQ and alert.

## DLQ and retry defaults

- DLQ requirement: keep the original event envelope + processing error details.
- Retry cap: start with 5 attempts and exponential backoff.
- Alert when DLQ depth > 0 for 10 minutes.

## Delivery channels

Phase 2:
- Publish event envelope to Pub/Sub topic (single shared topic recommended first).
- Domain services subscribe with service-specific subscriptions.

Phase 1:
- Use `logging` publisher to validate schema shape in staging.

## Correlation and tracing

- Pass `correlationId` from ingress requests into producer events.
- Propagate trace headers to async handlers and include in metadata fields.

## Migration checkpoint

- Before service split, complete this contract and update payload examples in CI-backed event tests.
- Once producer and all consumers pass schema validation in staging, move `DOMAIN_EVENT_MODE`
  from `log` to queue mode.
